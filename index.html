1.1. Introducció del projecte¶

El projecte consisteix en dissenyar i implementar una pipeline de dades a AWS per processar les vendes diàries d’una cadena de botigues de roba repartides per diferents ciutats de Catalunya. Les dades utilitzades són sintètiques i han estat generades específicament per simular un escenari realista de vendes per botiga i dia.

Cada botiga genera un fitxer CSV diari amb les vendes realitzades durant la jornada. Cada fitxer conté informació de les vendes realitzades en una botiga concreta, com l’identificador de la botiga, la data, l’hora de la venda, el producte venut, el departament, la talla, el preu, les unitats venudes, el color i la descripció del producte.

L’objectiu del projecte és emmagatzemar aquests fitxers a AWS, validar-ne l’estructura, processar les dades i carregar-les en una base de dades relacional perquè quedin preparades per a l’anàlisi. Com a resultat final, les transaccions quedaran emmagatzemades en una base de dades Amazon Aurora compatible amb MySQL i en S3. Com a validació addicional, Amazon Athena permetrà consultar amb SQL els fitxers emmagatzemats a S3 i comparar-ne els resultats amb les dades carregades a la base de dades.

La pipeline s’inicia quan es puja un fitxer CSV a la carpeta entrada/ del bucket S3. Aquesta pujada activa una funció Lambda que valida el fitxer i el mou a validat/ o error/. Quan un fitxer queda a validat/, S3 envia un missatge a una cua SQS, que activa una segona funció Lambda encarregada de processar el fitxer i carregar les dades a RDS.

1.2. Tipus de dades utilitzades¶

Les dades utilitzades representen vendes de productes de roba. Per una banda, es disposa d’un fitxer productes.csv, que conté el catàleg de productes disponibles. Per altra banda, cada botiga genera un fitxer CSV de vendes amb les operacions realitzades durant un dia concret.

El fitxer productes.csv té els següents camps:

id_producte
departament
descripcio
preu

El camp id_producte és l’identificador únic del producte i correspon a una cadena alfanumèrica. El camp departament indica la categoria del producte, com home, dona, nen o nena. El camp descripcio conté una breu descripció del producte i el camp preu indica el preu unitari.

Cadascun dels fitxers de vendes té els següents camps:

id_producte

1.4. Justificació dels serveis AWS¶

1.4.1 Amazon S3¶

Amazon S3 s’utilitza com a sistema principal d’emmagatzematge de fitxers. Dins del bucket es creen diferents carpetes lògiques per separar les fases del procés:

entrada/
validat/
error/
processat/

La carpeta entrada/ conté els fitxers CSV originals pujats al sistema. La carpeta validat/ conté els fitxers que han superat la validació inicial feta amb Lambda. La carpeta error/ conté els fitxers rebutjats o que no compleixen l’estructura esperada. Finalment, la carpeta processat/ conté els fitxers que ja han estat processats. Com a proposta futura, els fitxers de processat/ es conservarien durant dos anys i després es traslladarien a S3 Glacier mitjançant una política de cicle de vida.

S3 és adequat perquè és un servei gestionat, escalable i molt utilitzat com a repositori de dades. També permet mantenir la traçabilitat del procés, ja que es poden conservar separadament els fitxers originals, els validats, els erronis i els processats.

1.4.2 AWS Lambda¶

AWS Lambda s’utilitza en dues parts diferents de la pipeline.

La primera funció Lambda s’encarrega de validar els fitxers CSV que arriben a la carpeta entrada/ del bucket S3. Aquesta funció llegeix el fitxer CSV, comprova que la capçalera tingui l’estructura esperada i valida els registres abans de permetre que continuïn dins de la pipeline. Si el fitxer és correcte, el mou a validat/; si detecta algun error, el mou a error/.

La segona funció Lambda s’encarrega del processament posterior. Aquesta funció s’activa a partir dels missatges de SQS, llegeix el fitxer validat, processa el contingut del CSV i carrega les dades a la base de dades RDS.

Lambda és adequada perquè és un servei serverless basat en esdeveniments. Permet executar codi només quan cal, sense mantenir una màquina encesa permanentment per validar o processar fitxers.

1.4.3 Amazon CloudWatch¶

Amazon CloudWatch s’utilitza per consultar els logs generats per les funcions Lambda durant la validació i el processament dels fitxers. Això permet veure quan s’ha executat cada funció, quin fitxer s'ha processat, si el procés ha estat correcte o si s’ha produït algun error.

1.4.4 Amazon SQS¶

Amazon SQS s’utilitza per desacoblar la fase de validació de la fase de processament. Quan un fitxer és validat i queda a la carpeta validat/, S3 envia un missatge a una cua SQS amb la informació del fitxer.

Aquesta cua permet que la segona funció Lambda processi els fitxers validats de manera desacoblada i controlada. D’aquesta manera, la validació i la càrrega a RDS no depenen directament l’una de l’altra, i la pipeline és més robusta davant errors puntuals.

SQS també facilita el control dels fitxers pendents de processar. Quan la Lambda de processament acaba correctament, SQS elimina el missatge associat de la cua.

1.4.5 Amazon EC2¶

Amazon EC2 s’utilitza per a tasques d’administració i proves. Des d’aquesta instància es pot connectar a RDS, crear les taules de la base de dades, carregar el fitxer productes.csv i executar consultes SQL per verificar que les dades s’han carregat correctament. EC2 també s’utilitza per generar la capa de Lambda amb les dependències necessàries per connectar-se a RDS.

EC2 també serveix com a entorn de suport per preparar fitxers, executar comandes de prova i validar la connexió amb la base de dades. No és el component principal de processament de la pipeline, ja que aquesta part queda automatitzada amb Lambda.

1.4.6 Amazon RDS¶

Amazon RDS s’utilitza com a base de dades relacional final. En aquest projecte s’utilitza MySQL. Les dades netes generades durant el processament es carreguen en dues taules principals:

productes
vendes

La taula productes conté la informació dels productes, com l’identificador, el departament, la descripció i el preu. La taula vendes conté les vendes processades, com la botiga, el producte, la data, l’hora i la quantitat venuda. La relació entre les dues taules es fa mitjançant el camp id_producte.

RDS permet fer consultes SQL per obtenir indicadors com el nombre total de vendes, les unitats venudes, les vendes per botiga, les vendes per producte o les vendes per departament combinant la informació de vendes i productes.

RDS és adequat perquè les dades finals són estructurades i tabulars, i perquè MySQL permet consultar-les de manera senzilla i coneguda.

1.4.7 Amazon Athena¶

Amazon Athena s’utilitza com a servei addicional d’anàlisi. No substitueix RDS, sinó que serveix per validar i fer consultes exploratòries directament sobre els fitxers processats de S3.

Això permet comprovar que el fitxer processat de S3 és coherent amb les dades carregades a RDS. Per exemple, es pot comparar el nombre de registres consultat amb Athena sobre processat/ amb el nombre de files carregades a la taula vendes de RDS.

Athena s’ha triat perquè és serverless, fàcil de provar, s’integra bé amb S3 i permet afegir una capa d’anàlisi sense complicar l’arquitectura.

1.4.8 Amazon VPC¶

La part de xarxa es gestiona amb Amazon VP i es distribueix en dues zones de disponibilitat i inclou subxarxes públiques i privades. RDS i EC2 es despleguen dins la mateixa VPC, en una configuració que permeti la comunicació entre ells. També és necessari que la Lambda de processament pugui connectar-se a RDS per carregar-hi les dades.

La idea és que RDS no estigui obert innecessàriament a internet i que només els components autoritzats puguin accedir-hi. Com que s’utilitza MySQL, RDS escolta habitualment pel port 3306.

A més, utilitzem un VPC Endpoint per a S3 per evitar que les comunicacions entre Lambda i S3 passin per internet públic. Això millora la seguretat i manté el trànsit dins la xarxa d’AWS.

1.5. Processament de dades i integració entre serveis¶

El processament de dades segueix un esquema ETL: extracció, transformació i càrrega.

A nivell d’integració, el flux és el següent: els fitxers CSV es carreguen a S3 entrada/; una primera Lambda valida el fitxer i el mou a validat/ o error/; quan el fitxer queda a validat/, S3 envia a SQS un missatge amb la informació necessària perquè la Lambda de processament recuperi el fitxer des de validat/; la segona Lambda consumeix el missatge, processa el CSV i carrega les dades a RDS MySQL; un cop finalitza correctament el processament, el fitxer es mou a S3 processat/. CloudWatch registra els logs de les funcions Lambda i Athena permet consultar els fitxers processats guardats a S3.

1.5.1 Extracció¶

La fase d’extracció comença quan els fitxers CSV de vendes arriben a Amazon S3, dins la carpeta entrada/.

Cada fitxer representa les vendes d’una botiga en una data concreta. Aquesta fase no modifica les dades, sinó que les deixa disponibles en el repositori inicial.

1.5.2 Validació inicial¶

Quan es puja un fitxer CSV a S3 entrada/, la primera funció Lambda s’activa mitjançant un esdeveniment de S3. La funció llegeix el fitxer, comprova que la capçalera tingui l’estructura esperada i valida els registres abans de permetre que continuïn dins de la pipeline.

Si el fitxer supera la validació, es mou a validat/; si es detecta algun error, es mou a error/. El resultat de l’execució i els possibles errors queden registrats a CloudWatch.

Quan un fitxer arriba a validat/, S3 envia a SQS un missatge amb la informació necessària perquè la Lambda de processament recuperi el fitxer des de validat/.

1.5.3 Processament¶

La transformació i càrrega principal es fa amb una segona funció Lambda, activada pels missatges de SQS. Aquesta funció consumeix el missatge, recupera el fitxer validat des de S3 i interpreta el contingut del CSV.

Les operacions principals consisteixen a convertir els camps data, hora i unitats als tipus adequats i preparar la inserció de les dades a la taula vendes. Durant la càrrega, el camp unitats del CSV s’insereix al camp quantitat de la taula vendes.

La relació amb la taula productes es controla mitjançant el camp id_producte i la clau forana definida a RDS, que impedeix inserir vendes associades a productes inexistents.

1.5.4 Càrrega¶

La Lambda de processament carrega les dades a RDS compatible amb MySQL. Quan la càrrega finalitza correctament, mou el fitxer des de S3 validat/ fins a processat/.

La càrrega a RDS permet conservar les dades en format relacional i consultar-les posteriorment amb SQL. Després d’una execució correcta de la Lambda, SQS elimina automàticament el missatge associat de la cua.

No s’implementa en aquesta pràctica, però es preveu conservar els fitxers de processat/ durant dos anys abans d’arxivar-los a S3 Glacier.

1.5.5 Consulta i verificació a RDS¶

Un cop les dades estan carregades a RDS, es poden executar consultes SQL des de la instància EC2 per comprovar que la càrrega s’ha fet correctament i obtenir indicadors bàsics.

Per exemple:

SELECT COUNT(*) AS total_vendes
FROM vendes;

SELECT *
FROM vendes
ORDER BY data DESC, hora DESC
LIMIT 5;

Aquestes consultes permeten verificar el nombre de registres inserits, revisar les vendes més recents i comprovar que les dades queden disponibles per a l’anàlisi posterior.

1.5.6 Validació addicional amb Athena¶

Athena consulta directament els fitxers de S3 processat/. Per fer-ho, es crea una taula externa que apunta a aquesta ubicació i interpreta els CSV processats com una única taula consultable amb SQL.

Per exemple, es pot executar una consulta de recompte:

SELECT COUNT(*) AS total_vendes_processades
FROM vendes_processades;

També es poden fer consultes agregades, com el total d’unitats venudes per producte o l’import total de les vendes:

SELECT id_producte, SUM(unitats) AS total_unitats
FROM vendes_processades
GROUP BY id_producte
ORDER BY total_unitats DESC;

SELECT SUM(preu * unitats) AS import_total
FROM vendes_processades;

En aquesta part es configura el bucket de S3 i la funció Lambda que valida els fitxers d’entrada. A més, es comprova que el desencadenador de S3 està funcionant correctament i que els logs de CloudWatch mostren la informació esperada.

  1. Crear un bucket a S3 per a emmagatzemar els fitxers d'entrada, els fitxers validats i els fitxers amb errors.
    • Obrir AWS Management Console i busca el servei S3 des del cercador de serveis.

En aquesta part s'utilitza una cua de SQS per a gestionar els fitxers CSV que han estat validats per carregar-los a una base de dades RDS. Els missatges de la cua s'utilitzen per a coordinar el processament i la inserció de dades. Quan un fitxer és validat i mogut a validat/, S3 envia un missatge a la cua de SQS amb la informació del fitxer. Una funció Lambda està configurada per a llegir els missatges de la cua, carregar les dades del fitxer CSV a una base de dades RDS i eliminar el missatge de la cua un cop processat. A més, es comprova que el desencadenador de SQS està funcionant correctament i que els logs de CloudWatch mostren la informació esperada.

Consulta dels fitxers CSV processats amb Athena (servei addicional)¶

En aquesta part s'utilitza Amazon Athena, com a servei addicional, per a consultar directament els fitxers CSV que han estat processats i emmagatzemats a la carpeta processat/ del bucket de S3. D'aquesta manera es pot validar que les dades processades per Lambda es poden analitzar mitjançant consultes SQL sense necessitat de carregar-les prèviament a una base de dades relacional.

Athena permet crear una taula externa sobre els fitxers CSV ubicats a S3 i executar consultes SQL sobre aquests fitxers. Aquesta prova complementa la validació feta amb RDS, ja que permet comprovar les dades directament al data lake de S3.

  1. Configurar la ubicació dels resultats de les consultes d'Athena:
  • Anar al cercador de serveis i obrir Athena.
  • Entrar a Query editor.
  • Si és la primera vegada que s'utilitza Athena, configurar la ubicació on es desaran els resultats de les consultes.
  • Indicar una ruta dins del bucket de S3, per exemple:
s3://amzn-s3-uoc-2026-bucket/athena-results/
  • Aquesta carpeta s'utilitzarà perquè Athena pugui escriure els resultats generats per les consultes SQL.
  1. Crear una base de dades a Athena per al projecte:
  • Des del Query editor d'Athena, executar la següent sentència SQL:
CREATE DATABASE IF NOT EXISTS botigues_athena;
  • Un cop creada la base de dades, seleccionar-la al desplegable de l'esquerra o executar:
USE botigues_athena;
  1. Crear una taula externa sobre els fitxers CSV processats:

Els fitxers CSV de vendes processades estan ubicats a la carpeta processat/ del bucket de S3:

s3://amzn-s3-uoc-2026-bucket/processat/

Per consultar-los amb Athena, es crea una taula externa que apunta directament a aquesta ubicació de S3:

CREATE EXTERNAL TABLE IF NOT EXISTS vendes_processades (
    id_botiga string,
    data string,
    hora string,
    id_producte string,
    departament string,
    talla string,
    preu double,
    unitats int,
    color string,
    descripcio string
)
ROW FORMAT SERDE 'org.apache.hadoop.hive.serde2.OpenCSVSerde'
WITH SERDEPROPERTIES (
    "separatorChar" = ",",
    "quoteChar" = "\"",
    "escapeChar" = "\\"
)
LOCATION 's3://amzn-s3-uoc-2026-bucket-ipc/processat/'
TBLPROPERTIES (
    "skip.header.line.count"="1"
);

Aquesta configuració indica a Athena que ha de llegir tots els fitxers CSV de la carpeta processat/ com si fossin una única taula SQL. També s'indica que la primera línia dels fitxers correspon a la capçalera i no s'ha de tractar com una fila de dades.

  1. Comprovar que Athena pot llegir les dades de S3:
  • Executar una consulta de recompte per verificar que Athena detecta les files dels fitxers CSV processats:
SELECT COUNT(*) AS total_vendes_processades
FROM vendes_processades;
  1. Validar les vendes més recents:

Per comprovar que les dades es poden ordenar i analitzar correctament, es pot executar una consulta semblant a la utilitzada amb RDS:

SELECT *
FROM vendes_processades
ORDER BY data DESC, hora DESC
LIMIT 5;
  1. Fer comprovacions agregades sobre les dades processades:

Per comprovar les unitats venudes per producte, s'executa la consulta següent:

SELECT id_producte, SUM(unitats) AS total_unitats
FROM vendes_processades
GROUP BY id_producte
ORDER BY total_unitats DESC
LIMIT 10;

Per comprovar el volum de vendes per botiga:

SELECT id_botiga, COUNT(*) AS num_vendes, SUM(unitats) AS unitats_totals
FROM vendes_processades
GROUP BY id_botiga
ORDER BY num_vendes DESC;

Per calcular l'import total venut:

SELECT 
    SUM(preu * unitats) AS import_total
FROM vendes_processades;

El nombre de connexions és molt baix, amb un màxim d’1 connexió en moments puntuals. La memòria lliure es manté disponible i les latències de lectura i escriptura són baixes, amb lectures generalment per sota d’1 ms i escriptures al voltant d’1–1,5 ms, amb un pic puntual de 2,9 ms.

Les accions més importants serien:

  1. Apagar EC2 quan no s’utilitza.

  2. Revisar RDS segons l’ús real.

  3. Separar ús operacional i ús analític.
    RDS pot quedar per a consultes operacionals i dades recents, mentre que S3 + Athena pot assumir l’anàlisi històrica.

  4. Valorar alternatives serverless si la càrrega és molt variable.
    Si la càrrega fos molt discontínua, es podria estudiar Aurora Serverless. Si la pipeline creix en complexitat, Step Functions o Glue podrien ajudar a reorganitzar el processament.

  5. Diferenciar escalat vertical i escalat horitzontal.
    Si el problema és que una instància queda curta de CPU, memòria o connexions, es podria aplicar escalat vertical augmentant la classe de la instància RDS. 2.1 Script de Generació de Fitxers CSV de Prova¶

Aquest script en Python genera fitxers CSV amb dades de vendes simulades per a les botigues. Utilitza la biblioteca csv per escriure els fitxers i random per generar dades aleatòries. Els fitxers es desen localment a la carpeta data i després es poden pujar a S3 a la carpeta entrada.

L'script genera una base dades de productes i després crea fitxers CSV amb registres de vendes per a 30 botigues. Si es torna executar l'script es poden generar fitxers amb dades de vendes diferents reutilitzant els productes.

El fitxer productes.csv té la següent estructura:

  • id_producte: identificador únic del producte (cadena alfanumèrica de 12 caràcters)
  • departament: categoria a la qual pertany el producte (Ex. home, dona, nen o nena)
  • descripcio: breu descripció del producte
  • preu: preu unitari del producte

Cada fitxer de vendes té la següent estructura:

  • id_botiga: identificador de la botiga (Ex. BAR01, BAR02, LLE01, etc.)
  • data: data de la venda en format YYYY-MM-DD
  • hora: hora de la venda en format HH:MM:SS
  • id_producte: identificador del producte venut (ha de coincidir amb un id_producte del fitxer de productes)
  • departament: categoria del producte venut (ha de coincidir amb el departament del fitxer de productes)
  • talla: talla del producte venut (Ex. S, M, L, XL, etc.)
  • preu: preu unitari del producte venut (ha de coincidir amb el preu del fitxer de productes)
  • unitats: nombre de unitats venudes
  • descripcio: breu descripció de la venda (ha de coincidir amb la descripció del fitxer de productes)

2.2 Script de Lambda per a la validació dels fitxers CSV¶

Aquest script és una funció Lambda en Python que s’encarrega de validar els fitxers CSV que arriben a la carpeta entrada/ de S3. La funció es dispara amb un esdeveniment de S3 quan es puja un nou fitxer.

El codi llegeix el fitxer CSV, comprova que té la capçalera correcta i que cada registre compleix amb diverses validacions.

Si el fitxer és vàlid, es mou a la carpeta validat/. Si hi ha algun error, es mou a la carpeta error/ i es registra l’error a CloudWatch Logs.

El codi de la funció Lambda utilitza la biblioteca boto3 per interactuar amb S3 i la biblioteca csv per llegir els fitxers CSV.

2.3 Script de Lambda per al processament i càrrega a RDS¶

Aquest script és una funció Lambda en Python que processa els fitxers CSV que han passat la validació i que es troben a la carpeta validat/ de S3. Aquesta funció es dispara amb un esdeveniment de SQS, ja que quan s'afegeix un fitxer a la carpeta validat/, s'envia un missatge a una cua SQS que activa aquesta funció Lambda.

El codi llegeix el fitxer CSV, processa les dades i les carrega a la base de dades RDS compatible amb MySQL. Utilitza la biblioteca pymysql per connectar-se a la base de dades i executar les insercions. El codi també gestiona errors de connexió i d'inserció, a més es manté un registre dels resultats parcials a CloudWatch.

La biblioteca pymysql no està disponible per defecte a Lambda, per això s’ha creat una capa de Lambda que inclou aquesta biblioteca i s'ha adjuntat a la funció.

La biblioteca boto3 també s'utilitza per llegir els fitxers de S3 i moure'ls després de processar-los correctament. Si el processament és correcte, el fitxer es mou a la carpeta processat/.

Per a aquest script, és important configurar correctament les variables d'entorn per a la connexió a RDS, com ara el nom d'usuari, contrasenya, nom de la base de dades i l'endpoint.

2.4 Sentències SQL per a la creació de les taules a RDS¶

PREU_ALT = ["abric", "parka", "anorac"]
PREU_MIG = ["jaqueta", "dessuadora", "jersei", "texans"]


def preu_per_peça(peça: str) -> float:
    """Retorna un preu realista per a una peça de roba segons el seu tipus.

    Args:
        peça: El tipus de peça de roba (ex: "samarreta", "jaqueta", etc.)

    Returns:
        float: El preu realista per a la peça, amb decimals acabats en .95, .25, .75 o .50 per simular preus reals.

    Les peces d'abric, parka i anorac són les més cares, seguides de les
    jaquetes, dessuadores, jerseis i texans. La resta de peces tenen preus més
    assequibles.
    """
    # Es determina el preu base de la peça segons el seu tipus, amb rangs de
    # preus diferents per a les peces més cares, les de preu mig i les més
    # assequibles.
    if peça in PREU_ALT:
        preu = round(random.uniform(60, 120), 0)
    elif peça in PREU_MIG:
        preu = round(random.uniform(25, 60), 0)
    else:
        preu = round(random.uniform(10, 25), 0)

    # S'afegeixen els decimals al preu per simular valors reals, amb decimals
    # acabats en .95, .25, .75 o .50, segons la unitat del preu base.
    unitat = preu % 10
    if unitat == 9 or unitat == 4:
        preu += 0.95
    elif unitat == 0:
        preu += random.choice([0, 0.5])
    else:
        preu += random.choice([0.25, 0.75])

    return preu

def generar_hora() -> str:
    """Genera una hora realista per a una venda, amb més probabilitat durant les hores punta (11:30-12:30 i 17:30-18:30) i menys probabilitat en altres hores del dia.

    Returns:
        str: Una hora en format "HH:MM:SS".
    """
    r = random.random()

    if r < 0.25:
        inici = datetime.strptime("11:30:00", "%H:%M:%S")
        final = datetime.strptime("12:30:00", "%H:%M:%S")

    elif r < 0.50:
        inici = datetime.strptime("17:30:00", "%H:%M:%S")
        final = datetime.strptime("18:30:00", "%H:%M:%S")

    else:
        inici = datetime.strptime("09:00:00", "%H:%M:%S")
        final = datetime.strptime("21:00:00", "%H:%M:%S")

    # Es calcula un nombre aleatori de segons entre l'hora d'inici i l'hora
    # final, i s'afegeix aquest nombre de segons a l'hora d'inici per obtenir
    # una hora aleatòria dins de l'interval seleccionat.
    delta = final - inici
    segons = random.randint(0, int(delta.total_seconds()))
    hora = inici + timedelta(seconds=segons)
    return hora.strftime("%H:%M:%S")


def random_unitats() -> int:
    """Genera un nombre d'unitats venudes per a una venda, amb més probabilitat de vendre 1 unitat i menys probabilitat de vendre més unitats.

    Returns:
        int: Un nombre d'unitats venudes, entre 1 i 7, on 1 és el més probable i 7 el menys probable.
    """
    # Es defineixen els valors possibles (1 a 7 unitats) i els pesos associats a
    # cada valor, de manera que vendre 1 unitat és molt més probable que vendre
    # 7 unitats. Els pesos s'han ajustat per simular una distribució realista de
    # vendes, on la majoria de les vendes són d'1 unitat, seguides de 2 unitats,
    # i així successivament fins a 7 unitats, que són molt rares (el nombre de
    # peces es pot relacionar amb el nombre de fills d'una família, però no té
    # perquè ser així, s'ha agafat com a referència).
    valors = [1, 2, 3, 4, 5, 6, 7]
    pesos = [81, 15, 2, 1, 0.5, 0.3, 0.2]
    return random.choices(valors, weights=pesos, k=1)[0]

    botigues = []

        num = str(comptadors[ciutat_real]).zfill(2)

        botigues.append((f"{codi}{num}", ciutat_real))

def lambda_handler(event, context):
    """Funció Lambda que es desencadena quan es puja un fitxer CSV a la carpeta 'entrada/' del bucket S3.

    Args:
        event (dict): L'event que desencadena la funció Lambda, que conté informació sobre el fitxer CSV pujat a S3.
        context (object): L'objecte context que proporciona informació sobre l'execució de la funció Lambda.
    Returns:
        dict: Un objecte JSON amb l'estat de la validació i la ubicació del fitxer validat o amb error.

    La funció valida el format i les dades del fitxer CSV, i mou el fitxer a la
    carpeta 'validat/' si és vàlid o a la carpeta 'error/' si no ho és. Es
    retorna un objecte JSON amb l'estat de la validació i la ubicació del fitxer
    validat o amb error.
    """
    # Processar només el primer registre de l'event, ja que s'espera que cada
    # event contingui un sol fitxer CSV
    record = event["Records"][0]

    # Obtenir el bucket i la clau del fitxer CSV que ha desencadenat
    # l'esdeveniment
    bucket = record["s3"]["bucket"]["name"]
    key = record["s3"]["object"]["key"]

    # Es mostra un missatge de registre amb la ubicació del fitxer rebut per
    # facilitar el seguiment i la depuració.
    print(f"Fitxer rebut: s3://{bucket}/{key}")

    # Inicialitzar les variables per a la carpeta destí del fitxer un cop
    # validat i l'estat de la validació
    validated_key = ""
    is_valid = False

    # Llegir el fitxer CSV des de S3 i decodificar-lo com a text. Si hi ha un
    # error en la lectura del fitxer, es registra l'error i es retorna un estat
    # d'error.
    try:
        obj = s3.get_object(Bucket=bucket, Key=key)
        data = obj["Body"].read().decode("utf-8")
    except Exception as e:
        print(f"Error llegint el fitxer: {e}")
        return {"status": "error"}

    # Utilitza el mòdul csv per llegir el contingut del fitxer CSV i validar que
    # les columnes del fitxer coincideixen amb les columnes esperades. Si el
    # fitxer està buit o les columnes no coincideixen, es registra l'error i es
    # marca el fitxer com a no vàlid.
    csv_reader = csv.reader(io.StringIO(data))
    try:
        header = next(csv_reader)
    except StopIteration:
        print("Error: CSV buit")
        return {"status": "invalid"}

    if header != EXPECTED_COLUMNS:
        print("Error: format incorrecte")
        print(f"Esperat: {EXPECTED_COLUMNS}")
        print(f"Trobat:  {header}")
        is_valid = False
    else:
        is_valid = True

    # Si les columnes són correctes, es procedeix a validar cada fila del CSV.
    # Es comprova que cada fila té el nombre correcte de columnes, que els camps
    # no són buits i que els camps "preu" i "unitats" contenen valors numèrics
    # vàlids. També es valida que els camps "data" i "hora" contenen valors en
    # formats correctes. Si es detecta qualsevol error en les dades, es registra
    # l'error i es marca el fitxer com a no vàlid, i es deté la validació de les
    # files restants per evitar errors addicionals.
    if is_valid:
        for row_number, row in enumerate(csv_reader, start=2):
            if len(row) != len(EXPECTED_COLUMNS):
                print(f"Error: fila {row_number} té un nombre incorrecte de columnes")
                is_valid = False
                break

            values = dict(zip(EXPECTED_COLUMNS, row))

            # Validar que cada camp no és buit
            for column in EXPECTED_COLUMNS:
                if not values[column].strip():
                    print(f"Error: fila {row_number} té {column} buit")
                    is_valid = False
                    break
            if not is_valid:
                break

            # validar que preu és un número decimal més gran que 0
            try:
                preu = float(values["preu"])
                if preu <= 0:
                    print(
                        f"Error: fila {row_number} té preu no vàlid ({values['preu']!r})"
                    )
                    is_valid = False
                    break
            except ValueError:
                print(f"Error: fila {row_number} té preu no vàlid ({values['preu']!r})")
                is_valid = False
                break

            # validar que unitats és un número enter més gran que 0
            try:
                unitats = int(values["unitats"])
                if unitats <= 0:
                    print(
                        f"Error: fila {row_number} té unitats no vàlid ({values['unitats']!r})"
                    )
                    is_valid = False
                    break
            except ValueError:
                print(
                    f"Error: fila {row_number} té unitats no vàlid ({values['unitats']!r})"
                )
                is_valid = False
                break

            # validar el camp data es una data vàlida en format YYYY-MM-DD
            try:
                datetime.strptime(values["data"], "%Y-%m-%d")
            except ValueError:
                print(
                    f"Error: fila {row_number} té data no vàlida ({values['data']!r})"
                )
                is_valid = False
                break

            # validar el camp hora es una hora vàlida en format HH:MM:SS
            try:
                datetime.strptime(values["hora"], "%H:%M:%S")
            except ValueError:
                print(
                    f"Error: fila {row_number} té hora no vàlida ({values['hora']!r})"
                )
                is_valid = False
                break

    # Un cop finalitzada la validació, es determina la carpeta destí del fitxer
    # segons si el fitxer és vàlid o no, i s'imprimeix un missatge indicant si
    # el fitxer és vàlid o no. Si el fitxer és vàlid,
    if is_valid:
        validated_key = key.replace("entrada/", "validat/")
        print("CSV vàlid")
    else:
        validated_key = key.replace("entrada/", "error/")
        print("CSV NO vàlid")

    # Es mou el fitxer a la carpeta destí corresponent a l'estat de la
    # validació. Si hi ha un error en el procés en què es mou el fitxer, es
    # registra l'error i es retorna un estat d'error.
    try:

        s3.copy_object(
            Bucket=bucket, CopySource={"Bucket": bucket, "Key": key}, Key=validated_key
        )
        # Esborrar original
        s3.delete_object(Bucket=bucket, Key=key)

        print(f"Error movent el fitxer: {e}")
        return {"status": "error"}

    # S'imprimeix un missatge amb la ubicació del fitxer mogut per facilitar el
    # seguiment i la depuració.
    print(f"Fitxer mogut a: s3://{bucket}/{validated_key}")

    # Es retorna un objecte JSON amb l'estat de la validació i la ubicació del
    # fitxer validat o amb error.
    return {"status": "ok", "validated_file": validated_key}

    payload = json.loads(body) if isinstance(body, str) else body

    # Retorna el primer registre.
    return payload["Records"][0]


def lambda_handler(event, _context) -> dict:
    """Aquesta funció Lambda s'encarrega de processar els fitxers CSV que han estat validats i moguts a la carpeta "validat" del bucket de S3.

    Args:
        event (dict): Un event de SQS que conté informació sobre els fitxers CSV a processar.
        _context: Context de l'execució de la funció Lambda (no s'utilitza en aquesta funció).

    Returns:
        dict: Un objecte JSON amb l'estat de l'operació.

    La funció llegeix cada fitxer CSV, processa les dades i insereix els
    registres en una base de dades RDS. Un cop processat el fitxer, es mou a la
    carpeta "processat". Si hi ha algun error durant el procés, es registra
    l'error i es continua amb el següent fitxer.
    """
    # Crear un client de S3
    s3 = boto3.client("s3")

    # Llegir les variables d'entorn per a la connexió a RDS
    try:
        RDS_HOST = os.environ["RDS_HOST"]
        RDS_USER = os.environ["RDS_USER"]
        RDS_PASSWORD = os.environ["RDS_PASSWORD"]
        RDS_DB = os.environ["RDS_DB"]
        print("S'han llegit correctament les variables d'entorn per a RDS")
    except KeyError as e:
        print(f"Error: Variable d'entorn no definida: {e}")
        return {"status": "ok"}

    # Comprovar que Lambda resol correctament el DNS del RDS
    try:
        socket.gethostbyname(RDS_HOST)
        print(f"El DNS ha estat resolt correctament per RDS_HOST: {RDS_HOST}")
    except Exception as e:
        print(f"Error resolent DNS per RDS_HOST: {e}")
        return {"status": "ok"}

    # Connectar a la base de dades RDS
    print("Connectant a la base de dades RDS...")
    connection = pymysql.connect(
        host=RDS_HOST,
        user=RDS_USER,
        password=RDS_PASSWORD,
        database=RDS_DB,
    )

    # Comprovar que la connexió a RDS s'ha establert correctament
    if connection.open:
        print("S'ha establert correctament la connexió a RDS")
    else:
        print("Error en establir la connexió a RDS")
        return {"status": "ok"}

    # Processar cada registre de l'esdeveniment de SQS
    for record in event["Records"]:
        # Obtenir el missatge del registre SQS
        body = record["body"]
        print("Missatge rebut:", body)

        try:
            # Extraure la informació del registre S3
            s3_record = _extract_s3_record(record)

            # Obtenir informació del fitxer pujat
            bucket = s3_record["s3"]["bucket"]["name"]
            key = s3_record["s3"]["object"]["key"]
            print(f"Fitxer rebut: s3://{bucket}/{key}")

            response = s3.get_object(Bucket=bucket, Key=key)
            csv_content = response["Body"].read().decode("utf-8")
            csv_reader = csv.reader(io.StringIO(csv_content))
            next(csv_reader)  # Saltar la capçalera

            # Processar cada fila del CSV i inserir a RDS
            print("S'estan inserint dades a RDS...")
            with connection.cursor() as cursor:
                # Crear la consulta SQL per inserir les dades a la taula
                # "vendes". Aquesta consulta utilitza placeholders (%s) per
                # evitar vulnerabilitats d'injecció SQL i permet inserir les
                # dades de forma segura.
                for row in csv_reader:
                    values = dict(zip(EXPECTED_COLUMNS, row))
                    sql = "INSERT INTO vendes (id_botiga, id_producte, quantitat, data, hora) VALUES (%s, %s, %s, %s, %s)"
                    cursor.execute(
                        sql,
                        (
                            values["id_botiga"],
                            values["id_producte"],
                            values["unitats"],
                            values["data"],
                            values["hora"],
                        ),
                    )
                connection.commit()
            # Imprimir un missatge indicant que les dades han estat inserides
            # correctament a RDS.
            print("Les dades han estat inserides correctament a RDS")

            # Si s'ha processat correctament, moure el fitxer de la carpeta
            # "validat" a la carpeta "processat"
            processed_key = key.replace("validat/", "processat/")
            print("El fitxer CSV ha estat processat correctament")
            s3.copy_object(
                Bucket=bucket,
                CopySource={"Bucket": bucket, "Key": key},
                Key=processed_key,
            )
            s3.delete_object(Bucket=bucket, Key=key)
            print("El fitxer CSV ha estat mogut correctament")

        except Exception as e:
            print(f"Error processant el fitxer CSV: {e}")

    # Tancar la connexió a RDS un cop s'han processat tots els fitxers. És
    # important tancar la connexió per alliberar recursos i evitar possibles
    # problemes de connexió en futures execucions de la funció Lambda.
    connection.close()

    # Es retorna un objecte JSON amb l'estat de l'operació. En aquest cas, es
    # retorna "ok" encara que hi hagi hagut errors en el processament del
    # fitxer.
    return {"status": "ok"}

L'entorn de desenvolupament inclou les dependències de les funcions Lambda, però només per accedir ràpidament a la documentació i per a la gestió del codi. L'execució de les funcions Lambda es fa directament a AWS, on es poden configurar els desencadenadors, les variables d'entorn i les connexions amb altres serveis com S3, SQS i RDS.

Un dels maldecaps dels serveis d'AWS és la gestió dels rols i permisos. En AWS Academy Learner Lab no es poden crear rols, però es proporciona un rol preconfigurat (LabRole) amb els permisos necessaris per a les funcions Lambda i altres serveis utilitzats en el projecte. No obstant això, es proporcionen diversos exemples on es detallen els passos per crear rols personalitzats amb els permisos mínims o necessaris per a cada servei, com ara lectura i/o escriptura d'objectes de S3.

En les funcions Lambda és essencial seguir el flux d'execució del codi, per tal d'esbrinar on es produeixen els errors i com es poden solucionar. Per a això, s'han inclòs diversos punts de control i logs a CloudWatch per a cada funció Lambda, que permeten monitoritzar l'execució i detectar possibles errors o problemes de rendiment. Cal tenir en compte, que el comportament predeterminat de les funcions Lambda és que si es produeix un error, la funció es torna a executar automàticament i que s'ha de definir el temps d'expiració i el nombre màxim de reintents per a cada funció, per evitar bucles infinits en cas d'errors persistents.

Per a la generació de la capa de Lambda, hem tingut en compte que els dos membres de l'equip treballen en plataformes diferents (x86 i ARM), i per això hem optat per generar la capa de Lambda des de l'EC2. D'aquesta menera les instruccions per a generar la capa de Lambda són les mateixes per als dos membres de l'equip, i s'eviten problemes de compatibilitat entre les diferents arquitectures o captures de pantalla mostrant dues plataformes.

Hem decidit no utilitzar una IP pública per a l'RDS i configurar la base de dades des d'una instància EC2. Amb aquesta decisió es preten evitar exposar la base de dades a Internet i reduir el risc d'atacs externs.

Per a configurar la base de dades, primer s'ha d'iniciar una instància EC2 i connectar-se a ella mitjançant SSH. Un cop connectat, es pot utilitzar el client MySQL per a crear la base de dades, les taules i les relacions necessàries per al projecte. També es poden executar les consultes SQL per a validar que la base de dades està configurada correctament i que les dades s'estan carregant com s'espera. No s'ha assignat una adreça elàstica a l'EC2, ja que només s'utilitza puntualment per a la configuració inicial de RDS, i després es pot apagar per a estalviar costos.
